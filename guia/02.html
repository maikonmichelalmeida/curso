Curso RTL - Lab 02
Início Lab 01 Lab 02 Lab 03

Seu projeto, testado por módulo

Este lab pega o seu trabalho e transforma em uma rotina de verificação simples: testar blocos pequenos primeiro, depois testar o conjunto. Esse é um passo pequeno, mas já aponta para a organização dos labs e do ambiente do professor.

Ainda não estamos usando constraints, Tcl, Design Compiler, DC_NXT, Formality ou simulação gate-level. O foco aqui é só simulação RTL com VCS e visualização com Verdi.

1. O que mudou em relação ao Lab 01

No Lab 01 havia apenas um mux, um testbench, um filelist e um Makefile. No Lab 02 temos vários módulos do seu projeto: 

mux
alu
regbank
memory
control
top

A nova ideia é não testar tudo de uma vez. Primeiro testamos os blocos menores. Se eles funcionam isoladamente, fica mais fácil entender um erro no teste geral do top.

A pasta run/ não é fonte. Ela é saída de ferramenta. O Makefile cria uma subpasta para cada teste: 

simulation/run/mux/
simulation/run/alu/
simulation/run/regbank/
simulation/run/memory/
simulation/run/top/
Essa separação é um passo pequeno em direção ao ambiente do professor: fontes ficam em um lugar, resultados de execução ficam em outro.

3. Filelist: a lista do que o VCS compila

Cada teste tem seu próprio filelist. Por exemplo, o teste da ALU usa: 

filelist_alu.f

O conteúdo é:

../tb/tb_alu.sv
../rtl/alu.sv

O testbench vem primeiro para deixar claro que ele é o topo da simulação. O RTL vem depois porque é o DUT, isto é, o circuito sendo testado.

No teste do top, o filelist precisa incluir todos os módulos que o top instancia: 

../tb/tb_top.sv
../rtl/mux.sv
../rtl/regbank.sv
../rtl/alu.sv
../rtl/memory.sv
../rtl/control.sv
../rtl/top.sv

4. A variável TEST

O Makefile usa TEST para escolher qual teste rodar. Por padrão, TEST=mux. Então estes dois comandos são equivalentes: 

make test
make test TEST=mux

Quando você roda: 

make test TEST=alu

o Makefile monta estas escolhas:

Item Valor para TEST=alu
Filelistfilelist_alu.f
Top da simulaçãotb_alu
Pasta de execuçãorun/alu
Executávelrun/alu/simv
Waveformrun/alu/alu.fsdb

5. Comandos principais

Comando O que faz
make list Mostra os testes disponíveis: mux, alu, regbank, memory e top.
make show TEST=alu Mostra qual filelist, top, pasta de run e waveform serão usados.
make doctor TEST=alu Confere se VCS e Verdi estão disponíveis e mostra a configuração do teste.
make comp TEST=alu Compila com VCS e grava o simulador em run/alu/simv.
make sim TEST=alu Entra em run/alu e executa ./simv.
make test TEST=alu Executa a sequência: limpar, compilar e simular.
make waves TEST=alu Abre alu.fsdb no Verdi.
make all Roda todos os testes, um por vez.

6. Ordem recomendada no servidor

Primeiro atualize o repositório no SSH: 

cd ~/curso
git pull --ff-only origin main
cd 02/simulation

Se as ferramentas ainda não estiverem carregadas, carregue VCS e Verdi: 

module load vcs/W-2024.09-SP2-3 verdi/W-2024.09-SP2-6

Depois rode de forma gradual: 

make help
make list
make show TEST=mux
make doctor TEST=mux
make test TEST=mux

Se o mux passar, avance: 

make test TEST=alu
make test TEST=regbank
make test TEST=memory
make test TEST=top

Quando todos passarem individualmente, rode: 

make all

7. Onde olhar os resultados

Depois de rodar make test TEST=alu, veja: 

ls run/alu
less run/alu/comp.log
less run/alu/sim.log

Você deve encontrar algo parecido com:

run/alu/simv
run/alu/comp.log
run/alu/sim.log
run/alu/alu.fsdb
run/alu/simv.daidir/

O comp.log responde: "compilou?". O sim.log responde: "o teste passou?". A waveform responde: "os sinais fizeram o que eu esperava?".

8. O que observar em cada teste

Mux

Observe din1, din2, din3, din4, select e dout. A saída deve seguir a entrada escolhida por select.

ALU

Observe in1, in2, op, invalid_data, out, zero e error. O teste cobre soma, subtração, multiplicação, divisão, divisão por zero e dado inválido.

Regbank

Observe clk, rst, wr_en, din e dout. O valor só deve mudar na borda de clock quando wr_en está ativo, ou imediatamente no reset.

Memory

Observe memoryWrite, memoryRead, memoryAddress, memoryWriteData e memoryOutData. A escrita é sincronizada por clock; a leitura é combinacional.

Top

O teste do top é propositalmente simples. Ele verifica uma soma básica: 14 + 17 = 31. O objetivo não é provar o processador inteiro, mas confirmar que a integração compila e executa uma operação.

9. Por que esse passo está alinhado aos labs

Os labs da Synopsys usam Makefile, filelist, simulação, logs e waveform. O ambiente do professor também separa scripts, fontes e áreas de execução. O Lab 02 começa a trazer essa organização, mas sem colocar tudo de uma vez.

Se você consegue dizer qual filelist foi usado, qual top foi compilado, onde está o sim.log e onde está o fsdb, você entendeu o objetivo deste lab.
